Express vs NestJS: The Great Debate

Most Node.js tutorials start with Express. It's simple, unopinionated, and flexible. But "flexible" can quickly turn into "messy" in large projects. NestJS is a framework built on top of Express (or Fastify) that adds structure, similar to Angular or Java Spring.

The "Hello World" Comparison

Node.js + Express

You write everything manually. Routing, validation, and logic often get mixed up in one file.

const express = require('express');
const app = express();

// Route definition + Logic mixed
app.get('/users', (req, res) => {
  // Manual validation
  if (!req.query.id) return res.status(400).send('Error');
  
  // Database logic directly in controller
  const user = db.findUser(req.query.id);
  res.json(user);
});

app.listen(3000);
NestJS

Separation of concerns. Decorators handle the "plumbing".

@Controller('users')
export class UsersController {
  constructor(private userService: UserService) {}

  @Get()
  async getUser(@Query('id') id: string) {
    // Validation is automatic via DTOs
    // Logic is delegated to Service
    return this.userService.findUser(id);
  }
}

Why We Chose NestJS for Salex

Salex is a complex application with payments, scheduling, and messaging. We chose NestJS because:

  • Structure: It forces a modular architecture. You can't just put all your code in index.js.
  • TypeScript: It's built with TypeScript first. You get type safety out of the box.
  • Dependency Injection: Makes testing easy. We can swap the "Real WhatsApp Service" with a "Mock WhatsApp Service" without changing the controller code.
  • Decorators: @Get(), @Post(), @Body() make the code readable and declarative.

Anatomy of an Endpoint

Every API feature in Salex follows this pattern:

graph LR A[Request] --> B(Controller) B --> C{Service} C --> D[(Database)] C --> E[External APIs] D --> C E --> C C --> B B --> F[Response]
1. Controller (`*.controller.ts`)

The "Receptionist". It handles incoming HTTP requests, validates inputs, and sends the response. It should NOT contain complex business logic.

2. Service (`*.service.ts`)

The "Worker". It contains the actual business logic (e.g., "Calculate price", "Check availability", "Send email"). It interacts with the database.

3. Module (`*.module.ts`)

The "Container". It groups the Controller and Service together so NestJS knows how to assemble them.

API Walkthrough: Authentication

Let's look at real code from apps/api/src/modules/auth/auth.controller.ts.

Get Current User

This endpoint retrieves the profile of the logged-in user.

@Controller('api/v1')
@UseGuards(ConditionalAuthGuard) // <--- 1. Security
export class AuthController {
  constructor(private readonly userService: UserService) {}

  @Get('auth/me') // <--- 2. Route: GET /api/v1/auth/me
  async getCurrentUser(@Request() req: AuthenticatedRequest) {
    // 3. Access user from request object (injected by Guard)
    const userId = req.user.id; 
    
    return {
      success: true,
      data: { id: userId, ... },
      message: 'User profile retrieved'
    };
  }
}
Key Takeaway: Notice @UseGuards(ConditionalAuthGuard). This is a custom guard we built that allows us to switch between "Real Firebase Auth" and "Mock Auth" for easy testing.

API Walkthrough: Business & Routing

The core of Salex is the Business logic. See apps/api/src/modules/business/business.controller.ts.

Create Business

@Post() // <--- POST /api/v1/businesses
@UseGuards(ConditionalAuthGuard)
async createBusiness(
  @Request() req: AuthenticatedRequest,
  @Body(new ValidationPipe()) createData: CreateBusinessDto, // <--- Auto-validation!
) {
  // The controller just delegates to the service
  const business = await this.businessService.createBusiness(
    req.user.id, 
    createData
  );
  
  return { success: true, data: business };
}

Set Routing Code

This is a critical endpoint. It assigns the 4-digit code (e.g., "1234") to a business.

@Put(':businessId/routing-code')
async setBusinessRoutingCode(
  @Param('businessId') businessId: string, // <--- Extract URL param
  @Body() body: SetRoutingCodeRequest,
) {
  try {
    await this.businessRoutingService.setBusinessRoutingCode(
      businessId, 
      body.routingCode
    );
    return { success: true, message: 'Code set!' };
  } catch (error) {
    // Handle specific errors like "Code already taken"
    if (error.message.includes('already taken')) {
       throw new ConflictException('Code is taken');
    }
    throw error;
  }
}

API Walkthrough: WhatsApp Integration

This is where things get interesting. We have a Simulator controller for local dev.

Simulator: Send Message

apps/api/src/modules/whatsapp-simulator/whatsapp-simulator.controller.ts

@Post('sessions/:sessionId/send')
async sendMessage(
  @Param('sessionId') sessionId: string,
  @Body() request: { message: string },
) {
  // 1. Get the session state (are we in the menu? booking flow?)
  const session = await this.customerSessionService.getSession(sessionId);
  
  // 2. Route the message through our bot logic
  const response = await this.messageTypeRouter.routeMessage(
    { text: { body: request.message } }, // Mimic WhatsApp payload
    { currentState: session.currentState }
  );

  return { success: true, message: 'Message processed' };
}

Hands On: Your First Endpoint

Ready to write code? Let's say we want to add an endpoint to Get Business Reviews.

Step 1: Define the DTO

Create create-review.dto.ts to define what the user sends.

export class CreateReviewDto {
  @IsString()
  @IsNotEmpty()
  content: string;

  @IsNumber()
  @Min(1)
  @Max(5)
  rating: number;
}

Step 2: Update the Service

Add logic to business.service.ts.

async addReview(businessId: string, dto: CreateReviewDto) {
  return this.prisma.review.create({
    data: {
      businessId,
      content: dto.content,
      rating: dto.rating,
      createdAt: new Date()
    }
  });
}

Step 3: Update the Controller

Add the endpoint to business.controller.ts.

@Post(':businessId/reviews')
async createReview(
  @Param('businessId') businessId: string,
  @Body() dto: CreateReviewDto
) {
  return this.businessService.addReview(businessId, dto);
}
