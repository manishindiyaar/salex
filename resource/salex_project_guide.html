Welcome to Salex

A comprehensive guide to the Salex platform architecture, codebase, and development workflows.

Target Audience: This guide is designed for new developers joining the team. It assumes no prior knowledge of the codebase but expects familiarity with TypeScript, React, and Node.js.

What is Salex?

Salex is a platform that empowers salon businesses to manage their operations and appointments. Its unique selling point is its WhatsApp-first approach for customers, allowing them to book appointments simply by chatting with a bot, while merchants use a dedicated mobile app to manage their business.

Core Features

  • Business Routing: Unique 4-digit codes (e.g., 1234) that link customers to specific salons via a single WhatsApp number.
  • Dual-Mode WhatsApp: A Simulator for rapid testing and a Cloud API integration for production.
  • Merchant App: A React Native (Expo) app for salon owners to manage services, hours, and bookings.
  • Hybrid Auth: Supports both Firebase Auth (Production) and a Mock Auth system (Dev) for easy testing.

Prerequisites

Before you begin, ensure you have the following installed:

  • Node.js (v20.11.0 or higher)
  • pnpm (v8.0.0 or higher) - We use pnpm workspaces.
  • Docker - For running the local PostgreSQL database.
  • Expo Go (Optional) - On your phone if you want to test the mobile app on a real device.

Installation & Setup

1. Clone & Install

git clone https://github.com/your-org/salex-app.git
cd salex-app
pnpm install

2. Environment Setup

Copy the example environment files:

cp .env.example .env
cp apps/api/.env.example apps/api/.env
Important: Check apps/api/.env. To enable Mock Auth (no Firebase needed), ensure ENABLE_AUTH="false".

3. Database Setup

Start the database container and run migrations:

# Start Postgres
docker-compose up -d db

# Push schema to DB
pnpm db:push

4. Start Development Servers

We use TurboRepo to run everything at once:

pnpm dev

This will start:

  • API: http://localhost:3000
  • Merchant App (Expo): Starts Metro bundler (press 'i' to open in iOS simulator, 'a' for Android).

Component Breakdown

1. Merchant App (The Control Center)

Built with Expo & React Native. It is the interface for salon owners to manage their business. It does not store permanent data locally; it fetches everything from the API.

2. Salex API (The Brain)

Built with NestJS. It is the central source of truth. It handles business logic, validates bookings, manages the database, and processes WhatsApp webhooks.

3. WhatsApp Integration (The Customer Interface)

We do not build a customer app. Instead, we use WhatsApp as the UI for customers. The Salex API acts as a "Chatbot" that guides them through booking.

4. Database (The Memory)

PostgreSQL managed by Prisma. It stores Users, Businesses, Services, Bookings, and Chat History.

Data Synchronization & Workflow

How does data move between these components? Let's trace a typical booking flow.

1

Setup Phase

Merchant creates a business on the App. App sends POST /businesses to API. API saves to DB. Merchant sets a routing code (e.g., "1234").

2

Discovery Phase

Customer sends "Hi" to the Salex WhatsApp Number. Meta sends a Webhook to Salex API. API asks for the 4-digit code.

3

Booking Phase

Customer enters "1234". API looks up the Business in DB. API sends back the Service List via WhatsApp. Customer selects a service and time.

4

Sync Phase

API creates a Booking record in DB. Currently, the Merchant App polls the API (pull-to-refresh) to see this new booking.

Future Scope & Scalability 🚀

What else can be achieved with this architecture?

Real-time Updates: Currently, the app polls for data. We can implement WebSockets (Socket.io) so the Merchant App updates instantly when a booking comes in.
  • Push Notifications: Integrate Firebase Cloud Messaging (FCM) to alert merchants of new bookings even when the app is closed.
  • Multi-Tenancy: The current DB schema supports multiple businesses, but we can scale to sharded databases for millions of users.
  • Customer Web App: Since the API is decoupled, we can easily build a React Web App for customers who prefer not to use WhatsApp.
  • AI Analytics: Use the booking data to predict busy hours and suggest dynamic pricing to merchants.

Backend Architecture (NestJS)

The backend is built with NestJS, a progressive Node.js framework. It follows a modular architecture.

Key Technologies

  • Framework: NestJS
  • ORM: Prisma
  • Database: PostgreSQL
  • Validation: class-validator & class-transformer

Module Structure

Each feature is encapsulated in a module. For example, the BusinessModule contains:

  • business.controller.ts: Handles HTTP requests.
  • business.service.ts: Contains business logic.
  • business.module.ts: Registers the controller and service.

Authentication

We use a unique Conditional Authentication system to make development easier.

The Flow

graph TD A[Request] --> B{ENABLE_AUTH?} B -- True --> C[FirebaseAuthGuard] C --> D{Valid Firebase Token?} D -- Yes --> E[Pass] D -- No --> F[401 Unauthorized] B -- False --> G[MockAuthGuard] G --> H[Inject Mock User] H --> E

Mock Auth

When ENABLE_AUTH="false", the ConditionalAuthGuard bypasses Firebase verification and injects a hardcoded "Test User" into the request object. This allows you to test API endpoints without needing a valid Firebase token.

File: apps/api/src/modules/auth/conditional-auth.guard.ts

Database & Prisma

We use PostgreSQL with Prisma as the ORM. The schema is the source of truth for our data models.

Entity Relationship Diagram

erDiagram User ||--o{ Business : owns Business ||--|| Salon : has Business ||--o{ Service : offers Business ||--o{ Booking : manages Customer ||--o{ Booking : requests Customer ||--o{ WhatsAppConversation : participates Business { string id string routingCode json hoursOfOperation } Booking { string status datetime scheduledAt }

Key Models

  • User: The merchant/business owner (linked to Firebase UID).
  • Business: The core entity. Has a unique routingCode.
  • Customer: The end-user booking via WhatsApp. Identified by phone number.
  • Booking: Links a Customer, Business, and Service.

Business & Routing

The routing system is the "magic" that connects a WhatsApp message to a specific salon.

Routing Codes

Every business gets a 4-digit code (e.g., 1234). When a user sends "BOOK_AT_S1234", the system:

  1. Parses the code 1234.
  2. Lookups the Business with routingCode: "1234".
  3. Creates a session linking that Customer to that Business.

Smart Suggestions: If a merchant requests a code that is taken, the BusinessRoutingService suggests alternatives (e.g., if 1234 is taken, it suggests 1235, 1233, etc.).

WhatsApp Integration

We support two modes of operation:

1. Simulator (Dev/Test)

Located in apps/api/src/modules/whatsapp-simulator. This mimics the WhatsApp API behavior but runs entirely locally. It's used by the frontend to simulate chat flows without spending money on real messages.

2. Cloud API (Production)

Located in apps/api/src/modules/whatsapp. This handles real webhooks from Meta.

API Reference

This section documents all existing API endpoints. The API is versioned (currently v1) and uses standard HTTP status codes.

Base URL: http://localhost:3000/api/v1

Authentication Module

GET /auth/me

Get current user profile. Automatically creates user record on first call.

GET /auth/health

Health check for authentication system.

POST /auth/firebase/verify

Verify Firebase ID token and extract user details.

Business Module

POST /businesses

Create a new business.

GET /businesses/me

Get the current authenticated user's primary business.

GET /businesses/:businessId

Get public details of a specific business.

PUT /businesses/:businessId

Update business details.

PUT /businesses/:businessId/hours

Update business operating hours.

GET /businesses/:businessId/hours

Get business operating hours (Public).

GET /businesses/:businessId/open-status

Check if a business is currently open (Public).

Set the unique 4-digit routing code for a business (One-time).

Check if a routing code is available.

GET /public/businesses/by-code/:code

Find a business by its routing code.

GET /businesses/routing-codes/statistics

Get statistics on routing code usage (Admin).

Create a new service.

GET /businesses/:businessId/services

Missing / Planned APIs

The following features are core to a salon management system but are currently missing from the codebase.

Note: These are recommended additions for the next development sprint.

1. Staff Management

Currently, bookings are assigned to a business, not a specific staff member.

  • POST /businesses/:id/staff - Add a staff member.
  • GET /businesses/:id/staff - List staff.
  • PUT /businesses/:id/staff/:staffId - Update staff details/schedule.

2. Customer Management (CRM)

Merchants cannot currently view their customer list or history outside of bookings.

  • GET /businesses/:id/customers - List all customers who have booked.
  • GET /businesses/:id/customers/:customerId - View customer history/notes.
  • POST /businesses/:id/customers/:customerId/block - Block a customer.

3. Notifications & Settings

No endpoints to configure notification preferences (Email/SMS/WhatsApp).

  • PUT /businesses/:id/settings/notifications - Configure alerts.

4. Reviews & Ratings

No system for customers to rate their service.

  • POST /bookings/:id/review - Submit a review.
  • GET /businesses/:id/reviews - List business reviews.

5. Image Management

No endpoints to upload images for Business Logos, Cover Photos, or Service Images.

Workflow: Adding a New Feature

Let's say you want to add a "Staff Management" feature. Here is the step-by-step process:

1. Backend

  1. Schema: Add Staff model to schema.prisma. Run pnpm db:push.
  2. Module: Create apps/api/src/modules/staff.
  3. DTOs: Define CreateStaffDto in packages/shared-types so both frontend and backend can use it.
  4. Controller/Service: Implement CRUD logic.

2. Frontend

  1. API Client: Add methods to services/api.ts.
  2. Store: Add staffSlice to Zustand store if global state is needed.
  3. UI: Create StaffScreen.tsx.
  4. Navigation: Add route to MainTabNavigator or a Stack.

Troubleshooting

Issue: "Prisma Client not initialized"
Fix: Run pnpm db:generate in the api folder.
Issue: Expo app cannot connect to localhost
Fix: If testing on a real device, localhost won't work. You need to use your machine's LAN IP (e.g., 192.168.1.x) in the API client configuration.
