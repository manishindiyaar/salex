Database Architecture 🗄️

The backbone of Salex is a relational database (PostgreSQL) managed by Prisma ORM. This guide visualizes how data flows, how entities relate, and how we plan to scale to millions of users.

1. Business Perspective Model

This view focuses on the Merchant. How they manage their salon, services, and incoming bookings.

Key Flow: A User owns a Business. A Business has one Salon (profile) and offers multiple Services. Bookings are linked to a Business and a specific Service.

2. End-User Perspective Model

This view focuses on the Customer. How they interact with the system via WhatsApp or the Web.

Key Flow: A Customer is identified by their phone number. They create Bookings. They also have WhatsAppConversations which track their chat history with the bot.

Why Prisma ORM? 💎

We use Prisma instead of writing raw SQL queries. Here is why:

Type Safety

Prisma generates TypeScript types based on your schema. If you change a column name in the DB, your code will fail to compile immediately. No more runtime errors because of typos.

Migrations

Prisma Migrate manages your DB schema history. It generates SQL files automatically when you change `schema.prisma`, making it easy to deploy changes to production.

ORM vs. Raw SQL: A Comparison

What if we didn't use an ORM? Let's look at how we would fetch a Business and its Services.

Without ORM (Raw SQL)

You have to manually join tables and map the results to objects. It's error-prone and verbose.

SELECT b.*, s.* 
FROM "Business" b
LEFT JOIN "Service" s ON b.id = s."businessId"
WHERE b.id = '123';
-- Then you have to write JS to group services into an array

With Prisma ORM

Declarative and intuitive. Prisma handles the JOINs and data mapping for you.

const business = await prisma.business.findUnique({
  where: { id: '123' },
  include: { services: true }
});
// Result is a fully typed object with services array

Schema Implementation

This is the actual code in apps/api/prisma/schema.prisma that defines our models.

The Business Model

model Business {
  id          String   @id @default(uuid())
  name        String
  ownerId     String
  owner       User     @relation(fields: [ownerId], references: [id])
  
  // Relations
  salon       Salon?
  services    Service[]
  bookings    Booking[]
  
  // Unique Routing Code for WhatsApp
  routingCode String?  @unique
  
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt
}

The Booking Model

model Booking {
  id          String        @id @default(uuid())
  
  // Foreign Keys
  businessId  String
  business    Business      @relation(fields: [businessId], references: [id])
  customerId  String
  customer    Customer      @relation(fields: [customerId], references: [id])
  serviceId   String
  service     Service       @relation(fields: [serviceId], references: [id])
  
  startTime   DateTime
  endTime     DateTime
  status      BookingStatus @default(PENDING)
}

Future Proofing & Scalability 🚀

As Salex grows to millions of users, the database will become the bottleneck. Here is our roadmap for scaling.

1. Indexing (Immediate)

We add indexes to frequently searched columns to speed up queries.

model Customer {
  phoneNumber String @unique
  // Indexing phone number makes lookups O(log n) instead of O(n)
  @@index([phoneNumber])
}

2. Read Replicas (Growth Phase)

We can split the database into one Primary (for Writes) and multiple Replicas (for Reads). Since most app traffic is reading data (viewing services, checking slots), this offloads the main server.

graph TD App[API Server] Primary[(Primary DB)] Replica1[(Read Replica 1)] Replica2[(Read Replica 2)] App -- Writes (INSERT/UPDATE) --> Primary App -- Reads (SELECT) --> Replica1 App -- Reads (SELECT) --> Replica2 Primary -. Replication .-> Replica1 Primary -. Replication .-> Replica2

3. Sharding (Hyper-Scale Phase)

When the data is too big for one server, we Shard (partition) it. For Salex, we could shard by BusinessId or Region.

  • Shard A: Businesses in North India
  • Shard B: Businesses in South India

This requires significant application logic changes but allows infinite horizontal scaling.

Adding New Features (Migrations)

How do we add a new feature, like "Staff Members"?

  1. Edit Schema: Add model Staff { ... } to schema.prisma.
  2. Run Migration: Run npx prisma migrate dev --name add_staff.
  3. Generate Client: Prisma updates the TypeScript client automatically.
  4. Use it: prisma.staff.create(...) is now available in your code.
