# Story 1.6: Services Management (Backend Focus)

## Status
Ready for Review

## Story
**As a** salon owner, **I want** to create, manage, and organize my services (haircuts, styling, treatments, etc.) with pricing, duration, and descriptions through API endpoints, **so that** customers can see available services and book appointments with accurate pricing and time estimates.

## Acceptance Criteria
1. **Service Creation API**: POST endpoint allows business owners to create new services with name, price, duration, and optional description
2. **Service Retrieval API**: GET endpoints allow fetching all services for a business and individual service details
3. **Service Update API**: PUT endpoint allows business owners to update service details (name, price, duration, description)
4. **Service Deletion API**: DELETE endpoint allows business owners to remove services (with proper booking validation)
5. **Authorization Security**: Only business owners can manage their own services, with proper validation of business ownership
6. **API Testing**: Comprehensive test suite validates service management flows and business ownership security

### Backend Architecture Context [Source: architecture/12-backend-architecture.md]

**Service Module Structure:**
```typescript
@Injectable()
export class ServiceService {
  constructor(private prisma: PrismaService) {}

  async createService(
    businessId: string,
    ownerId: string,
    createServiceDto: CreateServiceDto
  ): Promise<Service> {
    // Validate business ownership
    await this.validateBusinessOwnership(businessId, ownerId);
    
    // Validate service data
    this.validateServiceData(createServiceDto);
    
    // Check for duplicate service names within business
    await this.checkDuplicateServiceName(businessId, createServiceDto.name);
    
    // Create service record
    return this.prisma.service.create({
      data: {
        ...createServiceDto,
        businessId,
      },
    });
  }

  async getBusinessServices(
    businessId: string,
    ownerId: string,
    pagination?: { page: number; limit: number }
  ): Promise<BusinessServicesResponse> {
    // Validate business ownership
    await this.validateBusinessOwnership(businessId, ownerId);
    
    // Get services with pagination
    const services = await this.prisma.service.findMany({
      where: { businessId },
      skip: pagination ? (pagination.page - 1) * pagination.limit : 0,
      take: pagination?.limit || undefined,
      orderBy: { createdAt: 'desc' },
    });
    
    // Calculate service summary
    const summary = await this.calculateServiceSummary(businessId);
    
    return {
      businessId,
      services,
      summary,
      pagination: pagination ? await this.calculatePagination(businessId, pagination) : undefined,
    };
  }

  async updateService(
    businessId: string,
    serviceId: string,
    ownerId: string,
    updateServiceDto: UpdateServiceDto
  ): Promise<Service> {
    // Validate business ownership and service existence
    await this.validateServiceOwnership(businessId, serviceId, ownerId);
    
    // Validate updated service data
    this.validateServiceData(updateServiceDto);
    
    // Check for duplicate names if name is being updated
    if (updateServiceDto.name) {
      await this.checkDuplicateServiceName(businessId, updateServiceDto.name, serviceId);
    }
    
    // Update service record
    return this.prisma.service.update({
      where: { id: serviceId },
      data: updateServiceDto,
    });
  }

  async deleteService(
    businessId: string,
    serviceId: string,
    ownerId: string
  ): Promise<void> {
    // Validate business ownership and service existence
    await this.validateServiceOwnership(businessId, serviceId, ownerId);
    
    // Check for existing bookings (future implementation)
    // const hasBookings = await this.checkServiceBookings(serviceId);
    // if (hasBookings) {
    //   throw new ConflictException('Cannot delete service with existing bookings');
    // }
    
    // Delete service record
    await this.prisma.service.delete({
      where: { id: serviceId },
    });
  }

  private async validateBusinessOwnership(businessId: string, ownerId: string): Promise<void> {
    const business = await this.prisma.business.findFirst({
      where: { id: businessId, ownerId },
    });
    
    if (!business) {
      throw new ForbiddenException('Access denied: Business not found or not owned by user');
    }
  }

  private async validateServiceOwnership(
    businessId: string,
    serviceId: string,
    ownerId: string
  ): Promise<void> {
    await this.validateBusinessOwnership(businessId, ownerId);
    
    const service = await this.prisma.service.findFirst({
      where: { id: serviceId, businessId },
    });
    
    if (!service) {
      throw new NotFoundException('Service not found');
    }
  }
}
```

### Service Validation Rules

**Input Validation Constraints:**
- Service Name: Required, 1-50 characters, alphanumeric with spaces and basic punctuation
- Price: Required, positive decimal, maximum $99,999.99, up to 2 decimal places
- Duration: Required, 15-480 minutes (15 minutes to 8 hours), must be divisible by 15
- Description: Optional, maximum 500 characters, basic HTML sanitization

**Business Logic Validation:**
- Unique service names per business (case-insensitive)
- Price must be greater than $0.01
- Duration must align with common booking intervals (15, 30, 45, 60, 90, 120 minutes)
- Service deletion prevented if active bookings exist

### Service Management Workflow

**Service Creation Flow:**
1. Business owner authenticates via ClerkAuthGuard
2. Business ownership validated for the specified businessId
3. Service data validated (name, price, duration constraints)
4. Duplicate service name check within business scope
5. Service record created with businessId foreign key
6. Service details returned for immediate use

**Service Management Best Practices:**
- Service names should be descriptive but concise (e.g., "Men's Haircut", "Deep Conditioning Treatment")
- Pricing should reflect market rates and include any additional costs
- Duration should account for setup/cleanup time beyond actual service time
- Descriptions should highlight key benefits or special techniques
