# Story 1.11: Remove Authentication from Customer-Facing API Endpoints

**Epic**: Foundation & Merchant Onboarding  
**Story ID**: 1.11  
**Priority**: Critical  
**Estimated Effort**: 3 story points  
**Status**: Ready for Review

## Story

**As a** WhatsApp customer using the simulator to interact with businesses,  
**I want** to view services, business hours, and available time slots without authentication errors,  
**so that** I can complete the booking workflow and test the WhatsApp simulator functionality end-to-end.

## Acceptance Criteria

### AC1: Customer Business Information Access
- WhatsApp customers can successfully GET business details (`/api/v1/businesses/:businessId`) without authentication
- WhatsApp customers can successfully GET business hours (`/api/v1/businesses/:businessId/hours`) without authentication  
- WhatsApp customers can successfully GET business open status (`/api/v1/businesses/:businessId/open-status`) without authentication

### AC2: Customer Service Information Access
- WhatsApp customers can successfully GET business services list (`/api/v1/businesses/:businessId/services`) without authentication
- WhatsApp customers can successfully GET specific service details (`/api/v1/businesses/:businessId/services/:serviceId`) without authentication

### AC3: Customer Time Slot Access
- WhatsApp customers can successfully GET available time slots (`/api/v1/businesses/:businessId/timeslots`) without authentication
- WhatsApp customers can successfully GET today's slots (`/api/v1/businesses/:businessId/timeslots/today`) without authentication
- WhatsApp customers can successfully GET week's slots (`/api/v1/businesses/:businessId/timeslots/week`) without authentication

### AC4: Business Owner Protection Maintained
- All business management endpoints (POST, PUT, DELETE operations) still require Clerk authentication
- Business analytics endpoints still require Clerk authentication
- Service management (create/update/delete) endpoints still require Clerk authentication

### AC5: WhatsApp Simulator Functionality Restored
- WhatsApp simulator shows actual services when clicking "📋 View Services" button
- WhatsApp simulator shows actual business hours when clicking "🕒 Business Hours" button  
- WhatsApp simulator shows available time slots when clicking "📅 Book Now" button
- No more 401 "Unable to load services" error messages in simulator

## Dev Notes

### Previous Story Insights
From Story 1.10, the WhatsApp simulator infrastructure is complete but hitting authentication barriers when customers try to access business data through the message routing system.

### Authentication Architecture Context
[Source: architecture/17-coding-standards.md#Authentication]
- All backend API endpoints that handle sensitive data or require logged-in user **must** be protected by `@UseGuards(ClerkAuthGuard)` decorator
- However, customer-facing read-only business data should be accessible without authentication to enable WhatsApp customer workflows

### Controller Structure and Guards
[Source: architecture/12-backend-architecture.md#Function Template]
- NestJS controllers use `@UseGuards(ClerkAuthGuard)` for securing endpoints
- Guards can be applied at controller level or individual endpoint level
- Individual endpoint guards can be removed while maintaining controller structure

### File Locations
[Source: architecture/10-unified-project-structure.md#modules]
- Business Controller: `apps/api/src/modules/business/business.controller.ts`
- Service Controller: `apps/api/src/modules/service/service.controller.ts`  
- Timeslots Controller: `apps/api/src/modules/timeslots/timeslots.controller.ts`

### API Endpoints Affected
[Source: architecture/05-api-specification.md#paths]
**Remove Authentication From:**
- `GET /api/v1/businesses/:businessId` - Business details
- `GET /api/v1/businesses/:businessId/hours` - Business hours
- `GET /api/v1/businesses/:businessId/open-status` - Open status
- `GET /api/v1/businesses/:businessId/services` - Services list
- `GET /api/v1/businesses/:businessId/services/:serviceId` - Specific service
- `GET /api/v1/businesses/:businessId/timeslots*` - Time slot endpoints

**Keep Authentication On:**
- All POST, PUT, DELETE operations
- All business management endpoints
- All analytics endpoints

### Data Models
[Source: architecture/04-data-models.md]
No changes to data models required - this is purely an authentication layer modification.

### Security Considerations
[Source: architecture/15-security-and-performance.md]
- This change exposes business profile data publicly but maintains write protection
- Future iteration should add rate limiting to prevent abuse
- Business-sensitive data (analytics, bookings) remains protected

### Testing

#### Testing Standards
[Source: architecture/16-testing-strategy.md#Backend Tests]
- **Integration Tests Required**: Test HTTP request/response flow for affected endpoints
- **Tools**: Jest & Supertest for API endpoint testing
- **Test Files Location**: `apps/api/test/` directory
- **Mocking**: Dependencies like Prisma client should be mocked in unit tests

#### Specific Testing Requirements for This Story
- **Integration Tests**: Create tests that verify endpoints work without authentication headers
- **Security Tests**: Verify business management endpoints still require authentication
- **End-to-End Tests**: Manual testing through WhatsApp simulator workflow
- **Network Tests**: Verify no 401 errors in API responses for customer endpoints
