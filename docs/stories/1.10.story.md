# Story 1.10: Comprehensive Interactive WhatsApp Simulator & Customer Session Management

**Epic**: WhatsApp Interactive Experience Foundation  
**Story ID**: 1.10  
**Priority**: High  
**Estimated Effort**: 20 story points  
**Status**: Tasks 1-12 Completed (Phases 1-4), Tasks 13+ Pending

## Story

**As a** product manager and developer building a production-grade WhatsApp booking system,  
**I want** a comprehensive interactive WhatsApp simulator with full customer session management and WhatsApp Cloud API interactive elements support,  
**so that** I can test complete booking workflows with buttons, lists, quick replies, and persistent customer sessions while maintaining the same codebase for both simulator and production WhatsApp integration.

## Acceptance Criteria

### Phase 1: Architectural Cleanup (Original Story Scope)

### AC1: Clean Production Webhook Separation
- Production `/webhooks/whatsapp` endpoint contains zero simulator-specific logic or conditional branches
- All simulator logic moved to dedicated `/simulate-webhooks/whatsapp` endpoint
- No simulator headers, parameters, or mode detection in production webhook controller
- Production webhook maintains existing functionality for real WhatsApp Cloud API integration

### AC2: Proper Business API Router Implementation  
- Create `BusinessApiRouterService` that routes simulator requests through existing business API endpoints
- Simulator uses authenticated API calls instead of direct database queries via Prisma
- All 23-25 existing business endpoints properly utilized (business lookup, services, timeslots, etc.)
- Maintain proper authentication and authorization patterns for API routing

### AC3: Dedicated Simulator Module Architecture
- Complete `/simulate-webhooks/whatsapp` endpoint in simulator module
- Simulator webhook handler with identical payload structure as production
- Independent configuration and environment handling for simulator mode
- Clean module boundaries with no cross-contamination between production and simulator

### Phase 2: Customer Session Management

### AC4: Customer Session Management (Option 2 Implementation)
- Implement `CustomerSessionService` with phone number as natural session key
- Create `CustomerSession` database table with phone-based session tracking
- Persistent conversation state across multiple interactions
- Session expiration and cleanup mechanisms
- Integration with existing `RedisSessionService` for performance

### AC5: Customer Authentication & Identity
- Phone number validation and normalization (international format)
- Customer creation and retrieval based on WhatsApp phone numbers
- Session linking to business context and conversation history
- Customer profile persistence across booking interactions

### Phase 3: Interactive WhatsApp Elements

### AC6: Full WhatsApp Cloud API Interactive Message Support
- Button messages (up to 3 buttons with IDs and text)
- List messages (sections, rows with IDs, titles, descriptions)
- Quick reply buttons (up to 3 quick replies)
- Menu/selection messages with multiple options
- Interactive message response handling (button_reply, list_reply)

### AC7: BusinessResponseTransformer Service
- Convert standard API responses to WhatsApp interactive message format
- Transform business services list into interactive list message
- Convert time slots into button selection messages
- Generate confirmation flows with interactive elements
- Template-based message generation for reusable patterns

### AC8: Message Type Router Enhancement
- Handle `text`, `button_reply`, `list_reply`, `quick_reply` message types
- Route different message types to appropriate business logic handlers
- State machine for complex multi-step booking interactions
- Context-aware response generation based on conversation state

### Phase 4: Enhanced Simulator UI

### AC9: Interactive Elements Rendering
- Render WhatsApp-style buttons in simulator chat interface
- Display list messages with proper section/row formatting
- Show quick reply options as clickable chips
- Interactive element clicking generates proper webhook payloads
- Visual feedback for button presses and selections

### AC10: Production-Grade UI Components

### Customer Session Management Architecture
**CustomerSession Table Schema:**
```sql
CREATE TABLE "CustomerSession" (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  phoneNumber VARCHAR(20) NOT NULL UNIQUE,
  businessId UUID REFERENCES "Business"(id),
  conversationState JSONB DEFAULT '{}',
  currentStep VARCHAR(50) DEFAULT 'welcome',
  sessionData JSONB DEFAULT '{}',
  expiresAt TIMESTAMP NOT NULL,
  createdAt TIMESTAMP DEFAULT NOW(),
  updatedAt TIMESTAMP DEFAULT NOW()
);
```

### Business API Integration Points
The BusinessApiRouterService will integrate with these existing endpoints (Make sure these API endpoints are already built use existing business service endpoints):
1. **Business Discovery**: `GET /api/v1/businesses/routing/{code}` - Business lookup by routing code
2. **Services Management**: `GET /api/v1/businesses/{id}/services` - Service listing for interactive lists
3. **Time Slots**: `GET /api/v1/businesses/{id}/timeslots` - Available slots for button selection
4. **Booking Creation**: `POST /api/v1/businesses/{id}/bookings` - Finalize booking process
5. **Business Hours**: `GET /api/v1/businesses/{id}/hours` - Operating hours validation
6. **Analytics**: `GET /api/v1/businesses/{id}/analytics/daily` - Business performance metrics

### Interactive Message Flow Design
**Booking Workflow State Machine:**
```
Welcome State → Service Selection State → Time Selection State → Confirmation State → Completion State
     ↓                    ↓                       ↓                     ↓                ↓
Text: "Hi!"    →    List Message      →    Button Message    →    Text Confirmation  →   Booking Created
                 (All Services)         (Available Times)      (Summary & Confirm)
```

### Authentication & Security Patterns
- **Customer Sessions**: Phone number as natural identifier, no login required
- **Internal API Routing**: Service account JWT for BusinessApiRouterService
- **Production Webhook**: Existing ClerkAuthGuard protection maintained
- **Simulator Webhook**: Separate authentication for testing environment

### Performance & Caching Strategy
- **Session Caching**: Redis integration for CustomerSessionService performance
- **API Response Caching**: BusinessResponseTransformer caches transformed messages
- **Interactive Element Caching**: UI caches rendered components for faster interactions
- **Database Optimization**: Indexes on CustomerSession.phoneNumber and expiresAt

### Technical Constraints & Compliance
- **WhatsApp Cloud API Limits**: Max 3 buttons, max 10 list rows per section
- **Message Size Limits**: Interactive messages have specific payload size restrictions
- **Rate Limiting**: Simulator must respect existing API rate limits
- **Session Expiration**: Customer sessions auto-expire after configurable timeout
